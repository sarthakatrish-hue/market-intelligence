02What we're deploying — current vs target

CURRENT
LOCAL DEV · 1 USER
  • One Python file — server.py · 1,446 lines · single process
  • Filesystem storage — wiki/, raw/, state.json
  • No auth — anyone on localhost:5173 is the curator
  • Agent = single LLM call returning a JSON spec; Python interprets it
  • Ingest is synchronous — submit blocks for 30–60s
  • No email · no digest · no scheduled jobs
  • Manual python3 server.py + npm run dev in two terminals
  • Reachable only from the maintainer's laptop · requires VPN for Odyssey
TARGET
PRODUCTION · 2–5 USERS
  • FastAPI app split into modules — routes/, agent/, jobs/
  • Supabase — Postgres + Storage bucket + Auth (magic-link)
  • Two roles — leader (read-only) + curator (full access)
  • Agent = real loop via Claude Agent SDK · 7 tools · 5–15 iterations
  • Ingest is async — submit acks instantly, job runs in BackgroundTasks
  • Daily digest via Resend at 09:00 IST · nightly git snapshot at 03:00 IST
  • Auto-deploy on git push · Vercel + Render both
  • Public HTTPS · always-on (14-min keepalive) · leaders + curator

What is NOT changing

  • CLAUDE.md ships verbatim as the agent's system prompt — same file drives local Claude Code today and the hosted agent in production. Zero schema drift.
  • The 9 page types, mandatory Intel sections, parse anchors, citation discipline — all unchanged.
  • Wiki / queries separation stays — queries remain in browser localStorage, never written to the wiki.
  • Maker–Checker filter logic — Haiku scoring, three-band routing, adversarial Checker prompt — same.
  • The frontend's visual language and 9 page components — same code, just deployed elsewhere.
Net effect: the behavior of the system converges on what local dev already does. The shape changes — split into proper modules, hosted, multi-user, async, push-enabled. Nothing about what the wiki looks like or how the agent reasons changes.

The 7 tools — concrete code shape

  • Each is a plain Python function in agent/tools.py, decorated and registered with the Claude Agent SDK
  • The LLM is told they exist via their JSON schemas — it cannot call anything else
  • One file, ~250 lines total, easy to audit
# backend/agent/tools.py — sketch

@tool
def read_raw(path: str) -> str:
    """Download a raw source file from the storage bucket."""
    return storage.download("raw-sources", path).decode()

@tool
def write_page(path: str, frontmatter: dict, body: str) -> dict:
    """Create or update a wiki page. Validates schema, parses Intel section, UPSERTs."""
    validate_schema(path, frontmatter, body)
    intel = parse_intel_section(body)
    supabase.from_("pages").upsert({"path": path, "frontmatter": frontmatter,
                                        "body": body, "intel_section": intel}).execute()
    return {"status": "ok", "path": path}

# ...plus list_pages, read_page, upload_raw, append_log, mark_layer2

Triggers + Views — how derived state stays in sync

TRIGGER · sync_sources_on_pages_change
Fires on INSERT/UPDATE of pages when folder='sources'. Reads frontmatter, derives the matching sources row (headline, touches, filter_score), UPSERTs it. Agent calls one tool; DB handles consistency.
VIEW · user_daily_usage
Aggregates query_log + ingest_cost_log by user/day in IST. Powers pre-flight cost-cap checks and the /api/me "12 of 30 queries today" display. Single SELECT — no app-level aggregation.
VIEW · monthly_spend
Sums all cost rows for the current month in IST. Pre-flight check before every Anthropic call — blocks the request if the monthly hard kill threshold is hit.

09Cost caps, rate limits & circuit breakers

Six layers of defense — defense in depth

  • Budget-based guardrails protect total spend (daily / per-job / monthly)
  • Rate-based guardrails protect against short-window spikes (per-minute, concurrency)
  • Health-based guardrails halt the system if Odyssey degrades or costs go anomalous
  • Pre-flight checks fire before any Anthropic call · in-flight checks fire per iteration · post-call updates the state

The three NEW additions (everything else was already locked)

★ CIRCUIT BREAKER
In-memory state per Render container. Tracks Anthropic failures in a sliding 60s window. State machine: closed → open → half-open → closed. On open, all LLM calls return 503 immediately for 60s. Then a single probe call decides whether to re-close or stay open. ~30 lines of Python.
★ CONCURRENT INGEST SEMAPHORE
Database-level check before starting an agent loop: SELECT count(*) FROM ingest_jobs WHERE status='running'. If ≥ MAX_CONCURRENT_INGESTS, the job stays queued instead of starting. The next completing job picks it up. On Render free tier (~0.1 CPU), 2 concurrent is the practical max. ~10 lines of Python.
★ COST VELOCITY TRIPWIRE
Single SQL view: SUM(cost_usd) WHERE created_at > now() - interval '1 hour'. If above COST_VELOCITY_HOURLY_USD (default $5), halt all LLM calls and fire a Sentry alert. Operator must manually reset (env var bump or DB flag). Catches runaway loops, abuse, or bugs hammering the LLM. ~15 lines of Python + 1 SQL view.

Module layout in backend/agent/

backend/agent/
  ├── client.py            # Anthropic SDK · ALL calls go through `guarded_call()` below
  ├── tools.py             # the 7 tools
  ├── filter.py            # Maker–Checker
  ├── ingest_worker.py     # agent loop
  ├── query_worker.py      # single-call query
  └── guardrails/          # ★ NEW · 6 layers of defense
      ├── circuit_breaker.py # layer 1 · in-memory state machine
      ├── rate_limit.py      # layer 2 · per-minute window
      ├── semaphore.py       # layer 3 · concurrent ingest DB check
      ├── cost_caps.py       # layers 4 + 6 · daily + monthly
      ├── velocity.py        # layer 5 · hourly spend tripwire
      └── guarded_call.py    # wraps every Anthropic SDK call · runs all checks
Single entry point. Every Anthropic call in the codebase goes through guarded_call(...). That function runs all 6 pre-flight checks, executes the call, runs post-call accounting, and updates the breaker. No exceptions, no bypass paths — guardrails can't be skipped by accident.

What this DOES and DOESN'T do

Covered ✓
  • Total spend ceiling (per-user + monthly)
  • Short-window cost spikes (velocity + per-minute)
  • Anthropic / Odyssey degradation (circuit breaker)
  • Runaway agent loops (iteration cap + stuck detection)
  • Multiple curators submitting at once (concurrent semaphore)
  • Per-ingest cost runaway (token cap mid-loop)
Deferred to v2
  • Per-role tiered caps (leader vs curator have same limits in v1)
  • Adaptive caps based on user behavior
  • Anthropic prompt-cache awareness in spend calculation
  • Auto-pause on Sentry error-rate spikes (separate from circuit breaker)
  • Real-time cost dashboard for the operator (Supabase Studio is enough for v1)
Total v1 implementation cost: ~80 lines of Python across 6 small modules. Adds ~5ms latency per Anthropic call (mostly the SQL for daily-usage view + velocity check). Every check is idempotent and stateless except the circuit breaker, which is per-container in-memory state.
