A reference document for everyone working on (or asking about) the Scapia Market Intelligence agent. It answers a single question that keeps coming up: "if I already have an LLM and an operating spec, why do I need to deploy anything? Isn't it just API calls?" By the end you'll understand what an agent really is, why a server is non-negotiable, what a tool actually does, and where MCP fits in.
A note on terminology. Throughout this document we refer to CLAUDE.md as the agent's operating spec (or just "the spec"). Use this phrase when explaining the system to others — it's clearer than "instructions" (too vague) or "operational contract" (sounds legal). Technically, the spec is loaded as the LLM's system prompt, so engineers may also call it that. Both terms refer to the same file.

01The core misconception

The intuition: "I give Claude an operating spec, it reads my raw source, and writes the wiki page. That's just an API call. Why deploy a whole server?"

What's missing: the LLM cannot write files. It can only return text. Someone has to read that text and do the writing. That someone is the agent — and it has to live on a server.

02What an LLM actually is

An LLM API is a pure function: text in → text out. No filesystem, no internet, no database. Whatever you send it, you get back text. Nothing else changes in the world.

Look closely at the code in the diagram. The model produced a beautiful markdown page that follows the spec. And it sits as a string in your terminal. Your filesystem is untouched. Postgres is untouched. The wiki has not been updated.

03Two flows, two shapes — and the 7 tools

The system runs two completely different shapes of work depending on what the user is doing. Query is a single LLM call. Ingest is a multi-step agent loop. Both pull from the same pool of 7 tools — but they call them in very different patterns, which is why one fits on a serverless function and the other does not.

Diagram — Query vs Ingest, side by side

Zoom in: what one iteration of that loop actually looks like

What's happening inside that × 5–15 ITERATIONS box? Each iteration is one round-trip between the LLM and our backend. The LLM never reads or writes anything itself — it emits a JSON request, our backend runs the matching Python function, and the result loops back. Same pattern as Claude Code on your laptop today; in production, our backend takes the place of the Claude Code CLI.

One iteration — LLM emits a request, the backend executes it, the result loops back
The key realization: Take away the backend (or Claude Code, if you're running locally), leave only the API call, and your wiki never updates. The LLM is just emitting instructions. The backend is what makes those instructions real. Repeat this round-trip 5–15 times and you have one ingest.
Diagram — Where each flow lands relative to Vercel's time cap
Time cap = the maximum number of seconds Vercel lets a single function run before it forcibly kills it. Hobby: 10s · Pro: 60s.

The 7 tools the agent can use

ToolWhat it does
read_raw(path)Download a raw source file from storage
list_pages(folder?)List wiki pages with summary fields
read_page(path)Return one page's frontmatter + body + Intel section
write_page(path, fm, body)Create or update a wiki page. Validates schema, parses Intel section, UPSERTs. Rejects if mandatory section/anchors missing.
upload_raw(path, content)Capture user verbal context as a raw file
append_log(op, desc)Add to the append-only ingest log
mark_layer2(...)Mark a borderline source for curator review — writes zero pages

Tools not given: delete_page, run_sql, web_search, fetch_url, send_email, git_commit. Each absence is a misuse the agent literally cannot commit — because no such tool exists for it to call.

04The agent loop — walked through

The "loop" sounds intimidating but it's a 20-line Python while. Let's walk it line by line, because once you see it once, the whole system clicks.

def run_ingest(raw_path):
    messages = [{"role": "user", "content": f"Ingest {raw_path}"}]   # the seed message

    while True:
        response = anthropic.messages.create(
            system=CLAUDE_MD,                  # ← the spec, sent every iteration
            tools=[write_page, read_raw, …],   # ← what the LLM is allowed to ask for
            messages=messages,                 # ← growing conversation history
        )

        if response.stop_reason == "end_turn":
            return response.content        # ← LLM said "I'm done", exit the loop

        for tool_use in response.content:    # ← LLM asked for one or more tools
            result = TOOLS[tool_use.name](**tool_use.input)   ← our code runs
            messages.append(response.content)          # LLM's tool_use turn
            messages.append({"tool_result": result})  # our tool result turn
        # loop back: LLM sees the tool result and decides what to do next

Read top to bottom:

01
Seed the conversation. We start with one message — "Ingest this raw file." That's all the LLM gets to begin with. The spec (CLAUDE.md) is loaded as the system prompt; it tells the LLM what the wiki looks like, what page types exist, what the rules are.
02
Call the LLM. We pass the system prompt + the conversation so far + the list of tools the LLM is allowed to ask for. The LLM reads everything and decides: "I need to read the raw source first" → it emits a tool_use for read_raw(...).
03
Check if it's done. If the LLM emits end_turn instead of a tool, the ingest is finished. We return the final text and exit. Otherwise, we continue.
04
Execute the tool. The LLM asked for read_raw("raw/atlas.md"). Our Python function with that name runs — pulls the file from storage, returns the contents. This is the moment the LLM's instruction becomes a real action.
05
Feed the result back. We append the LLM's tool_use message AND our tool_result message to the conversation. Now the conversation history says: "LLM asked for X, here's X."
06
Loop. Back to step 2. The LLM sees the raw source content, reads it, then emits the next tool — maybe list_pages() to see what exists, then write_page(...) to create the entity page, then append_log(...) for the audit trail. Each iteration the LLM either asks for another tool or says end_turn.

That's the whole agent. A typical ingest is 5–15 iterations of this loop. The Claude Agent SDK packages this while for you (handles retries, streaming, tool schemas) — but conceptually it's just this.

Why the loop matters for deployment: a single ingest is 5–15 LLM calls back-to-back, with code running between each. Total time: 20–60 seconds per ingest. The while loop has to run somewhere that can stay alive for that long — which is why the deployment needs a persistent container, not a "fire and forget" function.

05What "deploying an agent" really means

Deploying an agent = writing our own tiny version of the Claude Code CLI, then running it on a publicly-reachable machine so that users (not just developers at a terminal) can trigger it.

06"What about MCPs? Don't they replace agents?"

Reasonable thought: if there's a Swiggy MCP, a GitHub MCP, a database MCP — and they give the LLM "hands" — then why deploy an agent at all?

Short answer: MCP changes where the tools live. It does not remove the need for an agent. The loop, the trigger, the auth, the context, and the storage all still need to be deployed.

Diagram 5 — The agent has six layers. MCP standardizes only one.

MCP is genuinely useful — it's a standard, like USB-C for tools. Anyone can publish an MCP server; any agent can plug in. We could reuse community tools instead of writing our own. But the agent itself doesn't disappear: the loop, the trigger, the context, and the storage are still ours to host.

One legitimate alternative path exists: users run Claude Desktop on their laptops, our wiki ships as an MCP server, and users query it through Claude Desktop's UI. We'd skip the FastAPI app — but we'd also give up the Vault, Intel Cards, the Brief format, and the curator workbench. For an internal tool where the UX is the product, that trade-off is wrong.

Mental model to lock in

MCP is a protocol for tools, not a replacement for the agent. It lets one tool be reused by many agents (good) — but the loop, trigger, auth, context, and storage still need to live in a deployed app.

Someone built a project that emails you when groceries run low and includes a Swiggy Instamart link to reorder. Whole thing lives on Vercel. It looks like it contradicts everything above — it doesn't. Here's what's actually happening:

Diagram 6 — The inventory project, end to end (total runtime: ~5 seconds)

Why this works on Vercel and our ingest doesn't

Inventory project — fits Vercel
  • 1 LLM call, not a loop
  • ~5 sec total runtime
  • Human does the actual ordering — AI just composes the email
  • No background work, no streaming, no state held between requests
Our wiki ingest — doesn't fit
  • 5–15 LLM calls per ingest (the loop)
  • 20–60 sec total runtime
  • Agent does the actual writing — schema validation, multiple page writes, log entries
  • Async + background job + curator polls later
The principle: an agent loop fits on Vercel if and only if the entire loop finishes inside the function's time cap. "Swiggy MCP" isn't the magic — the magic is that the work is a single short pass. Take that same project and ask it to autonomously place orders, track status, retry on failure, and reconcile receipts, and you'd hit the wall instantly.
One-liner you can say

"That project works on Vercel because it's a 5-second cron function: check stock, write an email, include a deep link. It's not really an agent loop — it's a cron job with an LLM in the middle. Our ingest is 30–60 seconds of back-and-forth tool calls, which is structurally different and needs an always-on process."

07The one-line explanation

An LLM is text → text. Wiki pages need code that executes the LLM's instructions. That code (the agent) — the trigger, the loop, and the tool functions — has to run on a publicly-reachable server. The operating spec (CLAUDE.md) is the system prompt that tells the LLM how to behave. MCP can move tool definitions out of our codebase, but the loop that drives the LLM stays ours to host.
If someone asks: "why deploy an agent — isn't it just API calls?"

The LLM returns instructions, not actions. Something has to read those instructions and actually write the wiki, run the validation, hit the database. That "something" is our deployed agent — the FastAPI app on a server. It's not a wrapper around the LLM; it's the hands that carry out what the LLM decides.
