Scapia Market Intelligence — Architecture Brief

What we have built · what we need to host it

Scapia Market Intelligence is an LLM-maintained wiki tracking competitors, regulators, market signals, and partners. Curators feed raw sources; an LLM filter scores them; approved sources are written as structured wiki pages by Claude. Leaders ask plain-English questions and get a sourced "Brief" answer. The wiki is maintained by the LLM — not by humans.

2 Backend

2.a · Auth & Roles

What we have

  • No auth screen, no login, no users. The app runs on localhost — anyone with the dev URL has full access.
  • No role separation in code; the curator surfaces (/curator, /submit) are reachable by typing the URL.
  • No JWT, no session, no email flow.

What we need

  • Login screen — Sign in with Google, restricted to the @scapia.cards hosted domain (hd claim verified server-side; any other domain is rejected)
  • The backend runs the OAuth code exchange itself and issues its own session JWT (httpOnly cookie) — no third-party auth vendor
  • Three additive roles — a user may hold more than one; capabilities are the union:
    viewer (default, no role) = read Vault / Intel Cards / wiki / raw sources · leader = + ask questions (query) · curator = + submit & review (curator workspace) · admin = superuser + the Admin page (assigns roles)
  • Roles stored server-side (user_roles) and checked on every backend request — frontend gating is convenience only
  • Gated routes return 403: query endpoints for non-leaders, /submit + /curator/* for non-curators, /admin/* for non-admins
  • Admin page + access-request flow: a viewer hits a gate → clicks "Request access" → writes an access_requests row → flagged in the Admin page → admin grants/denies the role. Default admin seeded at deploy: sarthak.atrish@scapia.cards
  • A separate header-based secret (X-Internal-Secret) for cron-triggered endpoints — not exposed to users
User │ click "Sign in with Google" ┌────────────────┐ ├──────────────────────────────►│ Google OIDC │ ──► consent + redirect w/ code │ └────────┬───────┘ │ │ code ▼ ▼ Browser ◄── session JWT (httpOnly) ── Backend ── exchange code, verify hd=@scapia.cards │ │ upsert users row, look up user_roles │ every request: Cookie: session=<JWT> │ ▼ ▼ Backend ── verify JWT ──► union of roles ──► capability check ──► allow / 403 │ no capability ▼ viewer → "Request access" → access_requests (pending) ▼ Admin page grants role ──► user_roles row added

RBAC framework — what's already built (frontend, behind a swap-seam)

The split that matters: authorization (RBAC) is 100% ours and already built; authentication (Google login) is also ours to code but needs an externally-provided Google OAuth client. The whole RBAC layer — capability model, route guards, in-app gates, the access-request loop, and the Admin page — runs in the app today behind a deliberate pre-auth stub. It models the exact { name, email, roles[] } object the backend will hand the frontend after Google login, so wiring real auth changes one file and no consumers.

Capability model — one source of truth

  • Lives in src/auth/user.js. Roles are additive; capabilities are the union across a user's roles.
  • Helpers every surface calls: canQuery (leader|admin) · canCurate (curator|admin) · isAdmin · generic hasCapability(user, cap).
  • No role = baseline viewer (read-only). Admin is superuser.

Two enforcement points in the UI

  • Route guard — RequireCapability wraps /intelligence (query), /curator + /submit (curate), /admin (admin); a deep-link without the capability renders a full-page access card.
  • Inline gates — AccessGateToast on in-page actions: the query input, past queries, and the OPS Curator / Admin pills.

Access-request loop (built, end-to-end): a user without a capability hits a gate → Request access → writes a request row (today client-side mi_access_requests; the backend's access_requests table on production) → the Admin page shows it in a live pending queue → admin grants (adds the role) or denies. The Admin page carries its own dark sidebar and derives its user roster from real state only — the signed-in user plus anyone who actually filed a request — with no seeded/sample rows.

The swap-seam: only src/auth/user.js changes when real auth lands — its getUser() reads the session (the /api/me response / session JWT) instead of localStorage. Gates, the Admin page, the Sidebar and the route guards are untouched because they all go through useUser() and the capability helpers. Frontend gating stays convenience only: the backend re-checks roles on every request and returns 403 regardless of what the UI shows.

2.b · Agent layer (Ingest + Query)

What we have

  • Single Python file server.py (~1,400 lines) using stdlib http.server — fine locally, not production-grade
  • Filesystem reads/writes for wiki + raw + state
  • 10 HTTP endpoints (table below)
  • 5 LLM-driven roles in the pipeline (Haiku for filters/classifier, Sonnet for writers)
  • Synchronous: a submit blocks ~30–60s while the agent runs

What we need

  • FastAPI rewrite — async event loop (needed for streaming answers), input validation, OpenAPI docs, middleware for auth and CORS, graceful shutdown
  • Migration to Claude Agent SDK — proper tool-use loop, retries, structured tool errors
  • Background-task runner for ingest (submit returns immediately; the agent runs async)
  • Server-sent events (SSE) streaming for query answers (first token under 1s)
  • Restart recovery — on boot, mark any jobs left running as failed

2.c · Storage & Database

What we have today — filesystem

v1-LLM-wiki/ ├── wiki/ ← LLM-written knowledge base (markdown files) │ ├── index.md ← master catalog of all pages (LLM maintains) │ ├── log.md ← append-only event log (LLM maintains) │ ├── scapia.md ← reference baseline about Scapia itself │ ├── entities/ ← one file per competitor (Axis Atlas, CRED, HDFC…) │ ├── partners/ ← Federal Bank, Visa/MC/RuPay, OTAs │ ├── regulatory/ ← RBI circulars, DPDP, NPCI rules │ ├── events/ ← discrete market moments (launches, devaluations) │ ├── market-signals/ ← macro trends │ ├── concepts/ ← shared mechanics (FX markup, MDR…) │ ├── comparisons/ ← head-to-head, on demand │ └── sources/ ← one summary file per ingested source │ ├── raw/ ← immutable ingested source files (audit trail) │ ├── competitive/ ← Reddit, news, analyst notes │ ├── regulatory/ ← RBI circulars, MeitY notifications │ ├── market/ ← industry reports, traffic data │ └── ambiguous/ ← sources spanning multiple treatments │ ├── state.json ← pending Layer-2 queue, rejection log, health counts └── browser localStorage ← per-user query thread history, read state (orange dots)

What we keep — the filesystem, unchanged

Decision (locked): the markdown files stay the single source of truth — exactly as on the laptop today. index.md and log.md remain LLM-maintained files; they are core to the LLM-wiki architecture and are not migrated into database tables. The only change for production is where the files live: a shared object-storage bucket (so the separate service containers can all reach them) instead of one laptop's disk. PostgreSQL is added alongside the files — only for what a file cannot safely hold: identity, roles, job state, and per-user private chat.

The split — what is a file vs. what is a row

Lives as a file (object storage)Lives in PostgreSQL
All wiki/ pages — entities, regulatory, events, partners, market-signals, concepts, comparisons, sourcesUsers · roles · access requests (RBAC must not be world-readable)

scapia.md — reference baselineCost / usage records — tokens + USD per query and per ingest

Rule of thumb: the wiki (the "outside view") is files; everything operational, relational, private, or stateful is Postgres. The agent's file tools (read_page / write_page / read_raw) behave exactly as today — they just read and write against the bucket instead of local disk. Nothing about the CLAUDE.md schema, citation discipline, or index.md/log.md maintenance changes.

Reads & filtering — an in-memory index derived from the files

The Vault, Intel Cards, and live feed filter pages by folder, signal, domain, page-type, posture. Rather than denormalise those into a database table, the backend builds a small in-memory catalog from the files' frontmatter at startup and refreshes the affected entry on every write_page. At the current scale (low hundreds of pages) this is sub-millisecond and removes any risk of a file and a database copy drifting apart. A materialised pages_index table is a known, deferred optimisation — adopted only if the page count grows into the tens of thousands, and even then it stays a pure derived cache: the files remain source of truth.

PostgreSQL — operational data only (never the wiki itself)

users — one row per authenticated org account

Columns: id uuid PK · email UNIQUE (must end @scapia.cards) · name · picture_url · google_sub (OIDC subject) · status (active | disabled) · created_at · last_login_at

user_roles — additive roles (a user may hold more than one)

Columns: user_id → users · role (leader | curator | admin) · granted_by → users · granted_at · PK(user_id, role)

No row = baseline viewer (read-only). A user's capabilities are the union across their roles. Default admin seeded at deploy: sarthak.atrish@scapia.cards.

access_requests — the "Request access" → admin-grant flow

Columns: id uuid PK · user_id → users · requested_role (leader | curator) · reason · status (pending | granted | denied) · decided_by → users · created_at · decided_at

One open request per user per role. A pending row is what raises the flag in the Admin page's request queue.

ingest_jobs — async job state machine + full agent trace (the API → worker hand-off)

Columns: id uuid PK · raw_path · submitted_by → users · status · agent_trace jsonb · result jsonb · error · created_at · updated_at · completed_at

Status values: queued · running · completed · layer2 · rejected · failed · cost_capped

Replaces today's state.json queue. The API writes a queued row; the ingest-worker container polls, runs the agent, and updates status — no message broker needed. (The wiki event log stays in log.md; this table is job plumbing, not the audit log.)

query_threads — groups a user's follow-up questions into one chat (moves chat history off browser localStorage, cross-device)

Columns: id uuid PK · user_id → users · title · created_at · updated_at

Index: (user_id, updated_at DESC)

query_log — every Q&A turn (doubles as the per-query cost record)

Columns: id bigserial PK · thread_id → query_threads · position int · user_id → users · question text · answer text · pages_read text[] · input_tokens · output_tokens · cost_usd · created_at

Indexes: (thread_id, position) · (user_id, created_at DESC)

ingest_cost_log — token + cost record per ingest job

Columns: job_id → ingest_jobs · user_id → users · input_tokens · output_tokens · cost_usd · created_at

subscribers & digest_log — daily email digest

subscribers: id · email · name · paused · created_at

digest_log: id · sent_at · subscriber_email · source_slugs text[] · provider_message_id (idempotency)

Feed & catalog — derived from files, not a table

  • The live feed (today's sources idea) is built in-memory from the frontmatter of wiki/sources/*.md — no sources table, no trigger to keep in sync.
  • On every write_page, the backend re-parses that one file's frontmatter and updates its in-memory catalog entry. The file is always the truth; the catalog is a disposable cache rebuildable from disk at any time.

Row-Level Security (defense in depth)

  • query_threads + query_log: USING (user_id = current_user_id()) — a user can never read another user's chats even if the backend has a bug.
  • access_requests: a user sees only their own rows; only admin sees the full queue.
  • All other tables: read open to authenticated users; writes go through the backend's service-role key only.

Object storage — the wiki + raw, as files

  • One private bucket holding the working tree: wiki/ (all pages incl. index.md, log.md, scapia.md) and raw/ (immutable submitted sources).
  • Paths mirror today's layout exactly — wiki/entities/..., raw/competitive/... — so nothing in CLAUDE.md or the agent's tools has to change.
  • raw/ is write-once (audit trail); the agent's read_raw / upload_raw tools touch only this prefix. wiki/ is read-write via read_page / write_page.
  • Locally this is a mounted volume (or MinIO); in production a managed object store. The backend talks to it through one storage adapter, so local and prod differ only in config.

Cost & usage tracking — what gets stored on every LLM call

SurfaceStored whereFields
Every query turnquery_log (one row per turn)question · answer · pages_read · input_tokens · output_tokens · cost_usd
Every ingest jobingest_cost_log (one row per job)job_id · input_tokens · output_tokens · cost_usd
Filter calls (Maker · Checker · Intent)Folded into the parent job/query row (input/output tokens aggregated)(no separate table — keeps the cost surface simple)

Daily caps and monthly limits are computed from these two tables — no separate counters to keep in sync. The cost-cap pre-flight check is a single indexed SUM under 5 ms.

Query history saving — explicit

  • Today: stored only in the user's browser localStorage — lost on browser switch / clear cache.
  • Production: stored in query_threads + query_log scoped to user_id. Cross-device, durable, and the same rows power cost-cap aggregation.
  • RLS makes one user's chats invisible to every other user at the DB level — not just at the API layer.
  • Wiki pages are never joined to query tables — strict separation between "outside view" (wiki) and "inside view" (a user's chat).
