Query Intent Router

The query engine used to force every input through the Brief format — even "hi". This redesign classifies intent first and dispatches to the right response shape: 5 intents, 3 card formats, 0 wasted LLM calls on greetings.

The problem

▼ BEFORE

One template for everything. "hi" → full Intelligence Brief #4329 with Confidence chip, Blind Spot, and a Sonnet call burning ~15s.

Vague follow-ups like "tell me more" started from scratch instead of building on the prior turn.

▲ AFTER

Intent-routed responses. "hi" → friendly NoteCard in 10ms, no LLM. "tell me more" → ElaborationCard that anchors on the prior turn.

Substantive questions still get the full Brief. Out-of-scope inputs get a polite redirect.

Where it fits in the system

The intent router is a new sub-layer inside the read surface — it sits between request reception and response generation in /api/query. The wiki layer and raw sources layer are unchanged. Greetings and out-of-scope queries never reach the wiki context loader at all.

★ marks the new piece. Layer 2 (wiki) and Layer 1 (raw) are untouched. The intent router only decides which response generator to invoke — wiki schema, citation discipline, and the ingest workflow all behave identically to before.

Graceful degradation

If the Haiku classifier times out (Odyssey slow, VPN flaky), we fall through to brief_query — the original behavior. The user still gets an answer; only the cosmetic intent label is lost. Greetings keep working at full speed because they never touch the LLM.
